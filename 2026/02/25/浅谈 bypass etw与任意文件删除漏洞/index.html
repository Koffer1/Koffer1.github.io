六.任意文件删除漏洞

6.1.介绍

  • 主要聚焦于在 Windows 平台上 发现的常见文件处理逻辑错误，这类漏洞的独特性在于，它们无需逆向工程、模糊测试甚至编程能力即可被发现（实际上，往往只需使用 ProcMon 即可），但其潜在影响范围却极其广泛，从拒绝服务到权限提升。正因如此，它们被认为是一种“易于利用的漏洞”，且在安全研究领域中很有价值。发现这类漏洞的过程本身，就可以提供一套宝贵的基础侦察方法论。无论你的最终目标是什么，我始终建议将附加 ProcMon 并监控文件系统/注册表操作作为首要步骤。

6.2.攻击面

  • 漏洞的特性与目标：

    • 前提条件：漏洞通常为本地 (Local) 漏洞，要求攻击者具备基础系统访问权限。
    • 关键目标： 检查那些跨越权限边界（即在低权限用户和高权限（如系统/管理员）之间进行交互）的应用程序。
    • 重点关注对象： 驱动程序 (Drivers)、服务 (Services)、安装程序 (Installers)、启动应用程序 (Startup Applications) 以及自动提升权限的应用程序。
    • 攻击核心： 诱使一个具有特权的应用程序（如系统服务）在非预期的位置（通常是低权限用户可控的路径）执行文件系统操作。

  • 发现过程（ProcMon 应用）：

    • 环境准备： 启用 ProcMon (Process Monitor) 捕获所有活动。
    • 交互模拟： 模拟目标应用程序的行为（用户交互）。
    • 事件过滤： 调整 ProcMon 过滤器，重点审查文件系统和进程相关的 API 调用。

  • 特定场景举例：利用 Windows Installer (msiexec)

    • Windows 安装程序（Installer）是进行权限边界测试的经典目标，因为它在执行操作时拥有高权限。

    • 缓存目录： 安装程序相关的缓存文件通常位于 C:\Windows\Installer 目录下。

    • 触发高权限操作： 研究人员可以利用 msiexec.exe 的修复 (Repair) 功能来诱发高权限的文件系统操作。

      • 前提： 攻击者（非特权用户）可以执行此操作，因为修复是针对已安装程序的维护行为。

    • 触发命令示例：

  • 侦察目标： 观察当 msiexec 以高权限执行修复时，它在文件系统的哪些路径（特别是用户可控路径）进行了写入、创建或修改操作，这可能暴露权限处理上的逻辑缺陷。

6.3.示例

6.3.1.TOCTOU

  • 检查时间-使用时间 (TOCTOU) 错误，本质上是一种竞态条件 (Race Condition) 漏洞。
  • 当一个安全关键的应用程序，在对某一资源（如文件路径、内存地址或权限状态）执行检查（Time of Check, TOC）与执行最终使用（Time of Use, TOU）这两个步骤之间，缺乏原子性保证时，漏洞即产生。这种时间间隔为攻击者提供了机会，使其能够在检查完成和操作开始之间成功地篡改目标资源的状态。由于后续的操作（TOU）是基于一个已经被恶意篡改的、过时的状态信息执行的，程序行为将偏离预期，可能导致权限提升、数据损坏或拒绝服务等严重安全后果。

6.3.2.2.2.通过句柄设置文件信息:SetFileInformationByHandle
  • 此 API 也可以用于在已打开的文件句柄上设置删除信息 (Disposition Information)，这样该文件将在句柄关闭时被删除。

6.3.2.3.文件移动到任意写入
  • 文件移动操作的逻辑漏洞是本地权限提升（LPE）的常见攻击面。通过结合符号链接（Symlink）技术，攻击者可以劫持高权限进程（如系统服务）的文件重命名操作。例如，通过构造一个链式结构，将源文件的目标链接到恶意载荷（Payload），将目标文件（如备份文件）链接到目标服务二进制文件，即可在高权限上下文下实现恶意代码的替换。在 ProcMon 监控中，这类操作的底层对应 API 调用是 SetRenameInformationFile，是挖掘此类基于文件系统逻辑的 LPE 或任意文件读取漏洞的关键过滤项。
  • 如果文件移动到目标不可控但可读的位置，可以使用相同的逻辑进行任意读取操作。
6.3.2.4.结论
  • 文件写入漏洞更容易利用，只需覆盖一个不常用的 DLL/服务二进制文件即可获得系统控制权。
  • 文件读取漏洞则取决于目标设备。想想目标设备可能传输数据的方式。
    • 如果我们利用任意读取驱动程序的漏洞，并且该驱动程序将数据发送给用户模式客户端，我们就可以打开对客户端的句柄并读取其内存以获取数据。
    • 通过上述 ZDI 博客中提到的方法，可以利用文件夹创建漏洞来实现永久性的拒绝服务攻击，即创建名为“C:\Windows\System32\cng.sys”的文件夹。

7.4.利用安装服务进行本地权限提升

  • 当我们在 Windows 上利用任意文件删除时，会遇到三个问题：

    • 1.大多数关键的 Windows 操作系统文件都使用 DACL 锁定，甚至可以防止 SYSTEM 进行修改。相反，大多数操作系统文件归 TrustedInstaller 所有，并且只有该帐户有权修改它们。

    • 2.即使您找到可以删除的文件 SYSTEM ，它也必须是删除后会导致“打开失败”（安全性降低）的文件。

    • 3.由于共享违规，某些关键系统文件始终无法访问。

  • Windows Installer 服务负责执行应用程序的安装。应用程序作者提供一个 .msi 文件，该文件是一个数据库，定义安装应用程序时必须进行的更改：要创建的文件夹、要复制的文件、要修改的注册表项、要执行的自定义操作、执行，等等。

  • 为了确保在安装无法完成时保持系统完整性，并能够干净地恢复安装，Windows Installer 服务强制执行事务性。每次对系统进行更改时，Windows Installer 都会记录更改，并且每次使用正在安装的程序包中的较新版本覆盖系统上的现有文件时，它都会保留旧版本的副本。如果需要回滚安装，这些记录允许 Windows Installer 服务将系统恢复到原始状态。在最简单的情况下，这些记录的位置是名为 C:\Config.Msi 的文件夹。

  • 在安装过程中，Windows Installer 服务会创建一个名为 C:\Config.Msi 的文件夹，并在其中填充回滚信息。每当安装过程对系统进行更改时，Windows Installer 都会将更改记录在 C:\Config.Msi 内的 .rbs 类型文件（回滚脚本）中。此外，每当安装用新版本覆盖某些文件的旧版本时，Windows Installer 都会在 C:\Config.Msi 中放置原始文件的副本。这种类型的文件将被赋予 .rbf （回滚文件）扩展名。如果需要回滚不完整的安装，该服务将读取 .rbs 和 .rbf 文件，并使用它们将系统恢复到安装之前存在的状态。

  • 必须防止该机制被篡改。如果恶意用户能够在读取 .rbs 和/或 .rbf 文件之前对其进行更改，则在回滚期间可能会发生对系统状态的任意更改。因此，Windows Installer 对 C:\Config.Msi 及其所包含的文件设置了强 DACL。

  • 不过，这里出现了一个漏洞：如果攻击者具有任意文件夹删除漏洞怎么办？他们可以在 Windows Installer 创建 C:\Config.Msi 后立即使用它来完全删除它。然后，攻击者可以使用弱 DACL 重新创建 C:\Config.Msi （请注意，普通用户可以在 C:\ 的根目录下创建文件夹）。一旦 Windows Installer 在 C:\Config.Msi 中创建回滚文件，攻击者就能够将 C:\Config.Msi 替换为包含攻击者指定的 .rbs 和 .rbf 文件。然后，在回滚时，Windows Installer 将对系统进行任意更改，如恶意回滚脚本中指定的那样。
