2.3.3.1.利用 MSR 读/写操作在内核级执行代码

三.漏洞原语抽象

3.1 任意地址读写类

3.1.1.原语定义

  • 任意地址读写类原语，是指驱动允许攻击者控制内存访问操作中的关键参数，例如源地址、目标地址以及访问长度，从而对指定地址执行读取或写入。其底层实现方式并不固定，有的驱动会直接对用户提供的地址进行解引用，有的通过 memcpy、memmove、RtlCopyMemory 等拷贝接口完成数据搬运，也有的通过对 MmCopyVirtualMemory 进行不安全封装来实现跨地址空间复制。尽管实现路径不同，但从攻击者视角看，这类样本最终提供的能力是一致的，即读取目标地址中的内容，或将可控数据写入目标地址。
  • 结合前文样本分析，echo_driver.sys 与 dbutil_2_3.sys 虽然分别通过 MmCopyVirtualMemory 和 memmove 形成危险能力，但它们最终都可以收敛为同一种原语类型。也就是说，判断一条驱动路径是否属于任意地址读写类，关键不在于它调用了哪个具体 API，而在于攻击者是否能够稳定控制“读哪”“写哪”以及“读写多少”。

3.1.2.原语形式

  • 任意地址读写类原语可以抽象为如下形式：

  • addr 表示目标地址

  • size 表示读取或写入的长度

  • buf 表示用户态或攻击者可控缓冲区

  • 如果从更底层的数据复制视角去看，这类原语也可以统一表示为：

  • 当 src 为目标地址、dst 为用户缓冲区时，对应读操作；当 src 为用户缓冲区、dst 为目标地址时，对应写操作。对于一些驱动而言，读写能力并不是通过两个完全不同的函数实现的，而只是通过交换源地址和目标地址的角色来切换方向，因此这种“复制语义”在分析时也很重要。

3.1.3.典型实现与关键触发函数

3.1.4.原语价值与限制

  • 任意地址读写类原语是 BYOVD 场景中最核心、最通用的一类能力。读能力可以用于信息泄漏、内核对象定位、敏感字段提取以及关键地址探测；写能力则可以进一步用于修改内核对象、覆盖关键字段、篡改回调、修改 Token 或 patch 某些安全相关结构。很多完整的本地提权链，本质上都建立在一个稳定的读原语和写原语之上。
  • 不过，这类原语的能力边界并不总是完全对称。有的驱动只支持固定宽度访问，例如 1/2/4/8 字节；有的样本虽然看似同时具备读写能力，但其中一侧的地址或长度受到额外约束；还有一些驱动只能对特定地址空间起作用，例如目标进程地址空间、映射区域或某种包装后的缓冲区，而不是完全无约束的任意地址访问。因此，在分析这类原语时，除了确认是否“能读、能写”，还需要进一步判断地址是否完全可控、长度是否可控、方向是否可切换，以及访问结果是否能够稳定返回到用户态。
  • 从本文选取的样本来看，任意地址读写类最适合做半自动化模式提取，因为它通常伴随较为明显的危险路径特征，例如可疑的拷贝函数调用、用户可控地址参与解引用、或者 MmCopyVirtualMemory 这类跨地址空间复制接口的包装使用。这也是本文将其作为重点原语类别单独抽象的原因。

3.2.物理内存访问与映射类

3.2.1.原语定义

  • 物理内存访问与映射类原语，是指驱动允许攻击者指定物理地址、页描述信息或相关映射参数，并通过 MmMapIoSpace、MDL 或其他内存映射机制，将目标内存区域映射到内核可访问的虚拟地址空间，随后对该映射区域执行读取或写入操作。与普通任意地址读写不同，这类能力作用的对象通常不是虚拟地址，而是更底层的物理地址空间或由映射机制暴露出来的内存区域。
  • 从攻击者视角看，这类驱动并不一定直接提供“解引用任意地址”的能力，而是先把目标内存“接入”到可访问空间，再通过复制、写入或直接访问映射地址完成操作。因此，这类原语的本质通常不是单纯的 read/write，而是“映射 + 访问”的组合。结合前文分析，CVE-2025-8061 中对应的 LnvMSRIO.sys 就属于这一类：用户态可以通过 IOCTL 提供物理地址，驱动在内核中调用 MmMapIoSpace 建立映射，再通过包装后的拷贝函数对映射区域执行读取或写入。

3.2.4.原语价值与限制

  • 物理内存访问与映射类原语非常强，因为它绕过了普通虚拟地址语义，直接作用于底层物理页或映射区域。攻击者可以利用这类能力读取底层物理内存中的内容，或将用户提供的数据写入目标物理地址，从而间接影响内核中的关键结构。在某些场景下，这类原语甚至比普通任意地址读写更底层，因为它直接触及 RAM 的物理地址空间，而不是操作已经建立好的虚拟地址映射。

  • 不过，这类原语的利用门槛也更高。首先，能够访问物理地址并不意味着能够立即稳定修改任意目标对象，因为很多高价值目标最初是以内核虚拟地址形式存在的，攻击者仍然需要解决虚拟地址到物理地址的转换问题。其次，不同系统版本对物理页映射、PTE 访问以及相关内核机制的约束不同，一些在旧系统中可行的物理内存利用思路，在新版本 Windows 中未必仍然稳定有效。此外，映射类原语往往伴随对长度、缓存类型、页对齐等条件的要求，若驱动本身只支持固定粒度或特定映射大小，则能力也会受到限制。

  • 从本文的角度看，物理内存访问与映射类原语之所以重要，不仅因为它们本身危险，还因为这类驱动在代码层面通常具有较强的模式特征，例如 MmMapIoSpace、MDL 相关 API 与后续拷贝操作的组合，因而相对适合作为半自动化分析的目标类型之一。

3.3.CPU / 平台寄存器操作类

3.3.1.原语定义

  • CPU / 平台寄存器操作类原语，是指驱动将处理器或平台级寄存器的读写能力直接暴露给用户态，例如 MSR、控制寄存器（CRx）或 I/O 端口访问。与普通内存读写不同，这类能力操作的对象不是虚拟地址或物理内存，而是 CPU 本身的控制状态，因此往往具有更直接、更底层的系统影响。
  • 在 BYOVD 场景中，这类样本通常表现为驱动从输入缓冲区中解析寄存器编号、端口号或写入值，并直接调用 __readmsr、__writemsr、端口 I/O 或控制寄存器相关指令包装函数。若缺乏白名单、范围限制或权限校验，攻击者即可借此修改底层系统行为，例如读取关键寄存器状态、篡改系统调用入口，或影响某些处理器级保护逻辑。

3.3.4.原语价值与限制

  • CPU / 平台寄存器操作类原语的价值在于，它们作用于 CPU 的控制层，而不是普通内存层，因此一旦暴露，通常意味着攻击者能够直接干预系统底层行为。例如，MSR 读写可用于读取关键处理器配置，或在极端情况下修改系统调用入口等关键控制点；端口 I/O 则可能被用来与特定硬件或低层设备交互；控制寄存器写入更可能直接影响分页、保护位或执行环境。

  • 从自动化分析角度看，这一类的优势在于：关键 API 和指令包装模式非常集中，识别难度低于 UAF、对象生命周期错误或复杂回调劫持路径，因此非常适合作为 MCP 规则优先覆盖的一类能力。

3.4.进程 / 句柄 / Token 操作类

3.4.1.原语定义

  • 进程 / 句柄 / Token 操作类原语，是指驱动并不一定直接暴露传统意义上的内存读写能力，而是将对进程对象、进程句柄、访问令牌（Token）或相关安全属性的高危操作能力暴露给用户态。与任意地址读写类不同，这类原语更偏“对象级控制”，即攻击者直接操纵系统中的进程、句柄和安全上下文，而不是先通过内存原语间接实现同样效果。

4.3.面向 BYOVD 分析的提示词

4.3.1.为什么要设计提示词

  • 如果直接让大模型“分析这个驱动”，它很容易出现几个问题：

    • 大模型对复杂任务完成的效果不是很理想

    • 提示词可以给大模型提供思维链展示

    • 减少大模型输入长度

    • 大模型不好自主决策

  • 所以提示词的作用，不是让模型变聪明，而是给它一个固定分析路线，让它按你关心的方向走。

4.3.2.提示词设计原则

1.明确角色

  • 先告诉模型它现在是什么身份，比如：

    • Windows 驱动漏洞分析助手

    • 基于 IDA MCP 做逆向分析

    • 重点关注 BYOVD 风险路径

  • 这样做的目的，是让它从一开始就进入“驱动分析”状态，而不是泛泛地解释代码。

2. 明确任务拆分

  • 不要一句“分析这个驱动”，而是拆成几个小任务，例如：

    • 定位 DriverEntry / Dispatch

    • 枚举 IOCTL

    • 搜高危 sink

    • 回溯参数来源

    • 总结 primitive

  • 这样模型不容易乱跳，也更容易一步一步拿到有效结果。

3. 明确重点关注什么

  • 要直接告诉模型重点搜哪些函数和语义点，比如：

    • MmCopyVirtualMemory

    • MmMapIoSpace

    • __readmsr / __writemsr

    • ZwTerminateProcess

    • memcpy / memmove

    • EPROCESS / Token

  • 这样可以缩小搜索范围，减少它在普通逻辑里浪费时间

4. 明确输出格式

  • 如果不限制输出格式，模型每次回答都会不一样，后面很难对比和整理，所以最好提前要求它按固定字段输出，比如：

    • Driver

    • Dispatch

    • IOCTL

    • Call path

    • Sink

    • Constraints

    • Primitive class

    • Primitive form

4.3.3.提示词示例

  • 提示词本身也不一定全部由研究人员手工完成，随着大模型能力不断提升，研究人员完全可以先给出分析目标，再让模型自动生成一版提示词草稿，然后结合当前样本特点继续修改。

  • 这个示例的重点不是“写得多复杂”，而是把角色、步骤、重点函数和输出格式都说清楚。后面如果遇到特定类型的驱动，比如物理内存访问类、MSR 操作类、跨进程内存操作类，还可以在这个基础上继续补充更细的约束

  • 使用 vscode 打开一个目录并在里面创建AGENTS.md填写如下内容，即可使用了。

4.4.IDA MCP 功能扩展

  • IDA MCP 已经能提供函数、伪代码、交叉引用和调用链等基础信息，但这些能力本身仍然是通用的。对于 BYOVD 分析来说，真正耗时的往往不是“拿不到信息”，而是反复做入口定位、IOCTL 枚举、高危函数搜索和参数梳理等重复工作。因此，有必要进一步做面向场景的功能扩展，把这些高频操作交给脚本处理，从而降低重复劳动，提高风险点定位效率。

4.4.1.基于 IDAPython 的原型扩展

  • 先把重复性较强的筛选工作自动化，为后续的大模型辅助分析提供更干净的中间结果。由于使用IDA脚本很难匹配所有规则和全部样本场景，因此这里更偏向演示思路和辅助分析，而不是追求完全自动化。这里提到的都是大模型本身可以完成的，只是优化的时间，但是对于一些 混淆、加密识别等可以添加对应的功能。

五.实战挖掘与总结

  • 本来是想用 AI 直接挖出来一个品相比较好的，但是过程出现了一些问题，还是以总结为主吧，不过现在AI确实成为了分析的利器。 我找了近两年的一些以 Intel、NVIDIA、Microsoft 这类大厂驱动为例，代码质量、开发流程和外部审计通常更成熟，直接依赖 AI 进行静态分析，很难稳定挖出有利用价值的问题。相较之下，一些主板厂商及外围硬件厂商的驱动，例如某彩虹、某硕、某瑄等，更容易暴露出潜在风险点，但在最近两年的样本中，发现的问题仍以信息泄露、拒绝服务和稳定性缺陷为主，真正能够转化为高价值利用原语的漏洞并不多。

  • 相较于持续维护的一线厂商新驱动，小厂商、边缘硬件厂商、历史版本驱动，以及缺乏长期安全治理的产品线，更可能成为有效目标。换句话说，AI 的价值更接近于“加速筛选候选面”，而不是直接稳定产出可利用漏洞。

  • 我遇到最突出的问题是幻觉，以及同一样本在不同轮次分析中的结论漂移：同一个驱动、同一段路径、甚至同一个 IOCTL，模型在不同上下文下可能给出不一致的风险判断。AI 目前更适合作为辅助审计工具，而不是可靠的漏洞发现器。

  • 很多高价值使用场景本质上仍然依赖驱动摘链、关闭或绕过 EDR，而这类高防护环境对历史驱动、已知漏洞驱动、异常加载行为以及驱动签名状态的监控已经明显加强。旧驱动一旦加载就触发告警，新驱动则至少需要满足较新的签名要求，导致可直接利用的样本进一步减少。

  • 例如对 MmCopyVirtualMemory、进程结束、句柄操作、内核读写等高敏感能力的依赖，已经成为防护产品重点监控对象。即便驱动本身仍然暴露危险接口，只要利用行为落在“读写关键内核地址”或“结束 EDR 进程”这类强特征路径上，往往会在落地阶段直接失败。因此，今天的 BYOVD 不再只是“找到一个能 R/W 的驱动”那么简单，而是必须同时面对签名、加载、行为检测和后利用链条的多重约束。

  • 攻击空间相比过去已经被显著压缩。未来更有价值的方向，可能不再是单独追求一个驱动原语，而是把 BYOVD 看作整个内核利用链的一环，与更底层的内存破坏类漏洞结合，或者与更隐蔽的后利用技术结合。换句话说，真正稀缺的已经不是“能调起来的危险驱动”，而是“在现代防护环境中仍可稳定落地的利用链”。

  • 随着安全产品开始利用 AI 做行为关联、驱动加载画像和异常调用模式识别，依赖固定利用模板、固定敏感函数和固定攻击路径的 BYOVD 技术会越来越容易被识别。于是，攻击与防护之间的竞争重点，也会从“有没有漏洞”逐渐转向“利用链是否足够隐蔽、足够上下文感知、足够动态适配”。
