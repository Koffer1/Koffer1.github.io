5.2.内核堆风水

  • 每当听到“pool”这个词与 Windows 内核相关时，这些只是用于 Windows 内核堆管理的结构，主要关注两种分配器：常规分配器和大型/超大型池分配器。
    • 常规：用于任何适合在一页内的分配，这些分配利用空间来保存池头和初始空闲块。
    • 大内存：用于大于一页且占用一页或多页的任何数据。当使用内存池的全部内存时，无论分配大小如何，都会使用大内存CacheAligned type。如果不为分配的数据分配一整页，就无法轻松保证缓存对齐。
  • 因为在大型分配中没有空间放置页头，所以这些页面会存储在单独的“大池跟踪表”（nt！PoolBigPageTable）中。该表中的每个条目都由一个 POOL_TRACKER_BIG_PAGES 结构体来表示。

  • 需要注意的是，虚拟地址 (VA) 的值是通过逻辑或运算来指示其是否空闲或已被使用，最多只会有一个分配。

  • 最终，我们的目标是找到一个用户模式 API，使我们能够完全控制内核对象的内核模式数据，并创建一个大的池分配,以下是两个简单的例子：

    • 创建一个本地套接字，监听它，从另一个线程连接，接受连接，然后写入超过 4KB 的套接字数据，但不读取它。这将导致辅助功能驱动程序 (AFD.sys) 在内核模式内存中分配套接字数据。由于 Windows 网络协议栈在**DISPATCH_LEVEL (IRQL 2)**级别运行，且分页机制不可用，AFD 将使用非分页池缓冲区进行分配。
    • 创建命名管道，并发出大于 4KB 的数据写入请求，但不读取数据。这将导致命名管道文件系统 (NPFS.SYS) 也将管道数据分配到非分页池缓冲区中（因为 NPFS 也在 DISPATCH_LEVEL 级别执行缓冲区管理）。

  • 这两种方案中，方案二最简单，所需的代码行数也更少。需要注意的是，NPFS 会在缓冲区前面加上它自己的内部头部，称为DATA_ENTRY。每个版本的 NPFS 的 DATA_ENTRY 大小略有不同。

  • 解决这个问题的方法是创建具有正确偏移量的用户模式缓冲区。最后，关键在于缓冲区的大小至少要达到一页，这样我们才能强制使用大池分配器。
