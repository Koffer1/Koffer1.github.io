CVE-2024-26230

CVE-2024-26230

一.简述

  • CVE-2024-26230 是在 Windows 电话服务 (TapiSrv) 中发现的一个权限提升漏洞。该漏洞是因为在FreeDialogInstance函数中释放不当导致的UAF，通过伪造对象在TUISPIDLLCallback函数中获得代码执行权限。在上面的基础上，结合其他技术来绕过控制流保护（CFG）等缓解措施，进行加载dll最终通过PrintSpoofer 获取系统权限。

四.服务介绍

  • Windows 电话服务是 Windows RPC（远程过程调用）的一个组件，允许我们像与任何标准 RPC 服务一样与其进行交互。

4.1.接口定义语言（IDL）

  • IDL (Interface Definition Language) 是用于定义 RPC 服务器接口的语言。它的角色类似于 C 语言的头文件 (.h)，只负责声明接口的结构，包括可用的函数、参数和返回值类型，而不涉及任何具体的实现逻辑。这种”只声明，不实现”的特性，使其成为客户端与服务端之间的一份通信契约。因此掌握 IDL 是理解 Telephony 服务能力的前提。通过它，我们能清晰地了解 Telephony 提供了哪些接口，并知道如何正确地与服务进行通信。

五.漏洞成因

  • 在调度函数中，可以处理多个对象,这些对象由 NewObject 函数创建，该函数将它们插入到名为”ghHandleTable”的全局句柄表中。每个对象都持有一个独特的”魔数”。当电话服务引用某个对象时，它会调用 ReferenceObject 函数来比较”魔法”并从句柄表中检索它。

    • 一个用于保存”GOLD”对象
    • 一个用于调用调度函数”FreeDiagInstance”来释放”GOLD”对象，这样”GOLD”对象就被释放了，而原始句柄对象仍然保存着”GOLD”对象指针
  • 在TUISPIDLLCallBack函数中尝试使用已释放的GOLD对象（该对象现在是一个悬垂指针）。它访问v12 + 0x20并随后调用一个虚函数，从而触发UAF漏洞。

六.漏洞利用

6.1.对象伪造

在典型的 UAF 漏洞利用场景中，首先需要找到一个合适的对象，作为漏洞利用原语。在内核模式下，可以利用各种对象来实现此目的。然而，在用户模式下（TAPISrv以 RPC 服务的形式运行），该漏洞只能利用同一进程内的对象来利用。因此，如果目标进程中没有合适的对象，则无法直接利用该漏洞。

不在局限于寻找合适的对象作为漏洞利用原语，而是找到一个允许我们创建内存分配的原语，使其大小和内容都完全可控。通过这种方法，我们可以构造一个可以用来利用漏洞的伪造对象。其中一个原语是名为的调度函数TRequestMakeCall，它打开注册表项Software\\Microsoft\\Windows\\CurrentVersion\\Telephony\\HandoffPriorities并分配内存来存储键值。

通过检查注册表权限，确定该键可由当前用户完全控制，在函数”GetPriorityList”中，它首先检索键的类型和大小，然后分配一个堆来存储键值。这意味着，如果我能控制键值，我就能控制堆的大小和内容。”RequestMakeCall” 的默认类型是 REG_SZ，但由于当前用户对其拥有完全控制权限，可以删除默认值并创建一个 REG_BINARY 类型的键值。如上就可以将键值的大小和内容设置为任意值，使其成为一个实用的原语。

6.2.漏洞利用步骤

  • 使用上面描述的漏洞利用原语，可以通过以下设置利用 use-after-free 漏洞

    • 1.调用GetUIDllName创建一个GOLD对象。
    • 2.调用FreeDialogInstance释放GOLD对象，从而创建悬空指针。
    • 3.操纵注册表项值来为TRequestMakeCall使用后释放场景制作虚假对象。
    • 4.调用TUISPIDLLCallback以触发伪对象上的use-after-free，从而控制程序的执行流程。

6.3.对抗缓解措施

  • 经过上面的内存分配，控制键值伪造对象，当调用TUISPIDLLCallback函数触发”代码执行”时，控制缓冲区是RPC接口的输入缓冲区，如果我将其设置为3，它会将一个magic值与worker对象进行比较，然后从worker对象中获取一个虚函数地址，这样我就只需要在注册表键值的内容中设置这两个值即可。
  • 可以控制虚函数的地址，这意味着我可以控制 RIP 寄存器指向任何所需位置。然而，控制流保护 (CFC)缓解措施可以阻止在此次利用尝试中创建 ROP（面向返回编程）链。有关控制流保护的更多信息，请参阅以下参考资料：
    • 控制流防护缓解
    • 绕过控制流防护
    • 尽管如此，仍然可以调用二进制文件导入的 Win32 函数LoadLibraryW，实现加载被控制的DLL文件的步骤如下：
  • 1.利用上面描述的 UAF 漏洞调用malloc函数，如果malloc成功执行，在TUISPIDLLCallback函数中返回值的低 32 位会被写入 RPC 输入缓冲区并返回给客户端，因此导致信息泄露
  • 2.利用上面泄漏的低 32 位来调用VirtualAlloc，从而创建读-写-执行 (RWX) 内存区域
  • 3.将DLL路径复制到之前创建的RWX内存区域memcpy_s，由于复制操作每次限制为三个字符，因此我们需要重复此步骤几次
  • 4.使用上面复制到RWX内存区域的Dll路径，调用LoadLibrary函数，将生成一个具有权限的shell
  • 5.NT Authority/Network Service获得SeImpersonate权限。这是SeImpersonate一种 Windows 权限，它授予用户或进程模拟其他用户或帐户安全上下文的能力。此权限允许进程冒充其他用户的身份，使其能够像该用户一样执行操作或访问资源。但是，如果管理不当或授予未经授权的用户或进程，则SeImpersonate可能构成重大安全风险。在这种情况下，我们可以利用PrintSpoofer漏洞来获得更高的权限NT Authority/System。有关 PrintSpoofer 漏洞利用的更多详细信息，请参阅PrintSpoofer
