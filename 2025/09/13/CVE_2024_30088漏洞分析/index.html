CVE-2024-30088

CVE-2024-30088

一.前言

  • 该漏洞存在于NtQueryInformationToken函数中，特别是在处理AuthzBasepCopyoutInternalSecurityAttributes函数时，漏洞源于内核在操作对象时对锁机制管理不当，可能导致恶意实体的权限提升。

  • Windows 并没有启用 SMAP 保护，在启用 SMAP 保护的操作系统上，位于用户模式内存中的缓冲区必须使用特定的 API（例如 copyin 和 copyout ）复制到内核内存或从内核内存中复制出来。在 Linux 系统中，内核代码试图直接访问用户层内存会直接导致内存错误，而 Windows 内核被允许直接访问用户程序的数据，当然那个地址必须是有效的。

  • 许多内核函数在设计上并未考虑直接操作用户模式内存的复杂性与风险，通常基于一种”隐式信任”模型，即假设输入指针指向的是稳定且可信的内核内存，因此缺乏必要的异常处理和安全验证。为了安全地处理来自用户模式的内存，必须遵循一套严谨的多阶段方法，而不是单一的检查,比如地址合法性验证、数据一致性快照、并发访问控制等。本文讨论的漏洞正是这种情况：将用户指针，传递给应该只接受内核指针的函数。

三.漏洞成因

  • GitHub上已提供了一个公开的 POC，可以从普通用户到 SYSTEM 权限。当 NtQueryInformationToken 函数的 TokenInformationClass 字段设置为 TOKEN_ACCESS_INFORMATION 常量时，可以触发这个 bug（漏洞）。这个函数似乎完全无害，只是允许进程在拥有适当访问权限的情况下检索访问令牌的信息：
  • 这是一个读取函数，但这并不意味着我们不能用它实现任意写入
  • 对于 TOCTOU，跟踪非常重要，TokenInformation因为它是一个可由用户控制的缓冲区
  • 进入NtQueryInformationToken 函数，然后通过 switch case 查找常量TOKEN_ACCESS_INFORMATION，其SepCopyTokenAccessInformation是唯一一个以 in TokenInformation（用户提供的缓冲区）作为参数的函数。
  • 继续跟入SepCopyTokenAccessInformation之后，可以发现它在调用authzbasepqueryinternalsecurityattributestken来复制令牌对象的安全属性之前，会将一些关于令牌的数据复制到用户缓冲区中。*
  • 然后，authzbasepqueryinternalsecurityattributestken调用AuthzBasepCopyoutInternalSecurityAttributes，这就是可以找到错误的地方。
  • 当TokenObject->SecurityAttributesList中的_UNICODE_STRING结构体被复制到用户缓冲区时，会发生错误。
  • 具体来说，它是由于以下操作序列而出现的：
    1. 内核在用户缓冲区中存储一个指针pBuffer
    2. 内核将字符串复制到指向的内存中pBuffer
  • 在步骤 [1] 和 [2] 之间，我们可以使用竞争线程来更改pBuffer(内核写入的地址)，从而获得部分写入原语。
